# 产品流程与前端实现规划

## 1. 体验原则

NeTrix 的前端应呈现为一个具有智能辅助能力的学术社区，而不是 AI 工具控制台、校园行政门户或职业社交网站。用户进入后首先看到的应是学术帖子、资源、档案和连接机会；AI 是帮助学生理解自己、发现同伴、解释推荐的辅助层，而不是首页主角。

当前 UX 基线固定为：

```text
学术信息流优先 + 档案侧栏
```

桌面端以信息流为主轴，右侧档案侧栏承载档案进度、AI 建议、推荐档案和待处理请求。移动端不强行压缩侧栏，而应拆成清晰的主导航，例如 Feed、Library、Connections 和 Me。

## 2. 核心用户路径

第一版前端必须支持一条完整、真实、可重复的路径：

```text
认证 -> 引导 -> 信息流 -> 帖子详情或新建帖子 -> 档案优化
  -> 推荐 -> 请求连接 -> 接受连接后的消息
```

这条路径不是页面堆叠，而是一组状态变化。前端必须清楚表达邮箱是否已验证、档案是否完整、AI 是否正在生成、AI 输出是否仍为草稿、推荐是否可以请求连接、连接请求是否待处理，以及消息为什么可用或不可用。任何让用户误以为自己已经完成连接、AI 已经确认事实，或消息可以绕过接受流程的界面，都应视为产品错误。

## 3. 路由契约

第一版 App Router 路由应保持简洁稳定。前端负责人可以先建立路由骨架，后端与 AI 负责人随后接入真实数据和操作。

| 路由 | 责任 | MVP 要求 |
| --- | --- | --- |
| `/auth` | 登录、注册、校园邮箱验证入口。 | 使用 Supabase Auth 流程，不实现第三方社交登录。 |
| `/onboarding` | 建立基础档案。 | 收集专业、年级、课程模块、兴趣、可提供帮助、需要帮助和协作偏好。 |
| `/feed` | 主体验入口。 | 展示三类帖子、基础筛选、右侧侧栏或移动端入口。 |
| `/posts/new` | 创建帖子。 | 支持问答、资源、经验分享三种类型。 |
| `/posts/[id]` | 帖子详情。 | 展示作者学术昵称、档案摘要、标签和互动入口。 |
| `/library` | 资源中心。 | 展示精选资源和被提升展示的资源帖。 |
| `/me` | 当前用户档案。 | 展示已确认档案、AI 草稿、昵称和隐私控制。 |
| `/profiles/[id]` | 他人档案。 | 只展示可见性允许的字段和连接操作。 |
| `/recommendations` | 推荐页。 | 展示推荐档案、解释、请求和忽略操作。 |
| `/connections` | 连接管理。 | 展示待处理、已接受、已拒绝或已取消状态。 |
| `/messages/[threadId]` | 消息线程。 | 仅已接受连接的双方可访问。 |

`/feed` 是登录后的默认入口。若用户尚未完成引导，应跳转到 `/onboarding`，或显示明确的完成操作。不要让档案未完成的用户进入推荐流程。

## 4. 页面组成

信息流页面应以内容为核心，而不是解释产品概念。帖子卡片使用统一结构，通过帖子类型标签、颜色标记和元信息区分问答、资源和经验分享。不要为三类帖子建立三套完全不同的卡片系统。

引导页面应轻量但完整。推荐顺序是学术背景、课程模块与兴趣、技能与帮助需求、协作偏好、昵称建议、档案预览。AI 文案必须使用“建议”“草稿”“发布前请确认”这类语义，避免让用户感觉系统在替自己定义身份。

档案页和推荐页必须区分用户确认的事实与 AI 生成的草稿。推荐卡片展示“为什么这个人可能与你相关”，但不展示精确算法分数。连接按钮必须表达当前状态：可请求、已请求、已连接、已拒绝或不可操作。

消息页是闭环终点，不是开放私信系统。没有已接受连接时，前端应显示权限原因，并给出回到推荐或连接管理的下一步，而不是展示空消息输入框。

## 5. 组件约定

核心组件围绕业务对象建立，而不是围绕视觉碎片建立。第一版至少需要：

```text
PostCard
PostComposer
PostTypeFilter
AcademicProfileForm
NicknameSuggestionPicker
AcademicPortraitPanel
RecommendationCard
ConnectionRequestButton
ConnectionStatusBadge
MessageThread
LibraryItemCard
ProfileSidecar
```

组件接收明确的 DTO，不直接处理数据库原始对象。数据读取、写入操作、权限检查和 AI 调用通过服务端业务函数或 server actions 完成。组件文件可以处理界面状态，但不应包含评分逻辑、RLS 假设、提示词拼接或跨表查询。

共享界面基础组件放在 `src/components/`，业务组件放在 `src/features/<domain>/components/`。如果某个组件只服务于推荐功能，不应提前抽成通用组件。

## 6. 状态与反馈

第一版必须覆盖非理想路径。至少包括：

- 认证：加载中、邮箱未验证、链接无效、会话过期。
- 档案：未完成、保存中、AI 草稿生成中、AI 草稿生成失败、已确认。
- 信息流：无内容、筛选无结果、帖子创建失败、权限不足。
- 推荐：信号不足、生成中、已生成、已忽略、已发起请求。
- 连接：待处理、已接受、已拒绝、已取消、已经连接。
- 消息：线程加载中、消息发送失败、尚未接受连接、不是线程参与者。

状态反馈应告诉用户下一步能做什么，而不是只显示错误。界面文案应简洁、任务导向，避免用大段说明介绍产品功能或团队愿景。

## 7. 视觉与交互标准

视觉方向固定为干净、学术、温暖、具有社交感。界面可以有轻微智能感，但不能变成赛博或科幻风格，也不能模仿 LinkedIn 的职业社交语气，更不能像学校后台管理系统。

第一版需要集中定义设计变量：背景色、表面色、文本色、弱化文本、边框、强调色、帖子类型颜色、状态颜色、间距、圆角和阴影。Tailwind class 可以直接使用，但颜色、状态和间距的语义应统一。不要让每个页面自行发明一套视觉语言。

交互上应优先使用熟悉模式：用 tabs 或筛选标签处理帖子类型，用分段控件处理信息流和资源库切换，用徽标表达状态，用图标按钮处理轻量操作，用弹窗或抽屉处理确认类操作。桌面端侧栏不应遮挡信息流；移动端导航不应让用户迷失在过深页面层级中。

## 8. 前端验收标准

前端完成不等于页面看起来完整。合格前端必须在桌面和移动端跑通核心路径，必须处理加载、空状态、错误、权限不足、AI 生成中、AI 失败、请求待处理和已接受连接等状态。用户不应需要团队成员解释，也能理解自己当前在哪里、下一步能做什么、为什么某个操作暂时不可用。
