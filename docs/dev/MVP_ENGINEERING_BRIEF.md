# MVP 工程简报

## 1. 工程命题

NeTrix 第一版 MVP 要验证的工程命题很窄：学生是否能够通过公开学术内容产生可信信号，系统是否能够把这些信号整理成由用户确认的学术档案，并进一步转化为可解释、可请求、可沟通的学术连接。

因此，MVP 的成功不由页面数量或 AI 包装程度决定，而由一条真实闭环决定：

```text
已验证学生 -> 学术档案 -> 学术帖子 -> AI 辅助推荐 -> 连接请求 -> 接受连接后的消息
```

这条闭环必须使用真实后端、真实数据库、真实认证、真实 LLM API 和真实权限控制。静态样稿、假按钮、手动修改数据库或只在本地状态中伪造结果，都不能证明当前产品方向。

## 2. 已锁定技术基线

第一版技术路线已经锁定，不再由各成员自行选择：

```text
应用目录：apps/web
框架：Next.js App Router
语言：TypeScript
样式：Tailwind CSS
认证：Supabase Auth，配合校园邮箱验证
数据库：Supabase Postgres
查询与结构：Drizzle
AI 调用：服务端 LLM 封装层
包管理命令：corepack pnpm
```

这组选择服务于 MVP 阶段的现实约束：团队需要快速构建网页产品，需要校园邮箱验证，需要关系型数据与清晰权限边界，需要保护 AI 密钥，需要用 TypeScript 降低前后端契约漂移，并需要在短时间内形成可演示闭环。

Supabase 提供认证、Postgres 和行级安全策略；Drizzle 负责数据结构、迁移友好的类型和查询层；Next.js 同时承载页面、server actions、route handlers 和轻量业务服务。第一版不拆微服务，不引入独立后端框架，不引入复杂队列系统，也不构建原生应用。

## 3. MVP 范围

第一版必须覆盖认证、档案、帖子、AI 档案与昵称、推荐、连接请求，以及接受连接后的消息功能。Library/Resource Centre 是辅助入口，用来承接资源帖和精选资源，但不能抢占主产品身份。

第一版不包含以下内容：首页聊天机器人、外部 AI 记忆导入、复杂黑箱匹配、UNNC 全校超级应用、完整社团或活动系统、复杂积分声誉体系、多角色后台、课程答疑 AI，以及旧版本中偏赛博、网络节点或 Oracle 的视觉概念。

MVP 的功能判断原则是：如果某项功能不能直接强化帖子、档案、推荐、连接中的至少一环，就应放入后置范围。

## 4. 应用形态

代码应以单一网页应用起步，目录固定为 `apps/web`。这个选择保持当前实现简单，同时为后续加入数据库脚本、种子数据、端到端测试、文档和部署配置留下清楚边界。

第一版应用应形成五个能力面：

1. 身份与引导：校园邮箱认证、会话、档案引导、学术昵称确认。
2. 学术内容：三类帖子的创建、浏览、详情、筛选和种子内容。
3. 档案智能：用户确认的档案、AI 草稿或优化建议、画像状态。
4. 推荐与连接：候选生成、透明评分、LLM 解释、请求、接受和拒绝。
5. 消息与验证：仅限已接受连接的消息线程、事件记录和核心路径验收。

这五个能力面不是五个独立产品，而是同一条验证路径的连续环节。开发顺序应尽快把这些环节串起来，而不是先把某一个页面打磨到接近最终视觉。

## 5. 实施里程碑

第一轮开发按“基础能力 -> 内容信号 -> AI 辅助 -> 连接闭环 -> 演示加固”的顺序推进。

**里程碑 0：脚手架基线。** 建立 `apps/web`、Next.js App Router、TypeScript、Tailwind、`corepack pnpm` 脚本、`.env.example`、Supabase/Drizzle 基础文件和功能模块目录。

**里程碑 1：身份与档案。** 实现校园邮箱认证、会话保护、引导表单、档案编辑、昵称建议占位契约和档案完成状态。

**里程碑 2：帖子与信息流。** 实现问答帖、资源帖、经验分享帖的基础增删改查、信息流、详情页、帖子类型筛选、演示种子内容和空状态、错误状态。

**里程碑 3：AI 档案与推荐。** 接入服务端 LLM 封装层，实现昵称建议、学术画像优化、候选生成、透明评分、LLM 解释和推荐记录。

**里程碑 4：连接与消息。** 实现请求、接受、拒绝、连接状态转换、仅限已接受连接的消息线程、权限辅助函数和 RLS 校验。

**里程碑 5：质量与演示。** 补齐核心测试、移动端布局、错误状态、事件记录、演示用户、种子数据、核心路径脚本或手动验收说明。

## 6. MVP 验收标准

MVP 完成时，团队应能用 3-5 个演示用户重复演示完整路径：一个已验证用户完成档案，浏览或发布三类学术帖子，获得 AI 辅助档案或昵称，看到推荐档案和推荐理由，发送连接请求，对方接受后进入消息线程。

该演示不得依赖开发者手动修改数据库，不得绕过权限检查，不得使用真实学生个人数据作为演示内容。系统还应记录足够的行为事件，例如 `profile_completed`、`post_created`、`recommendation_generated`、`recommendation_clicked`、`connection_requested`、`connection_accepted`、`message_sent`。后续商业组和产品判断扩展价值时，应依赖这些行为信号，而不仅是团队主观感受。
