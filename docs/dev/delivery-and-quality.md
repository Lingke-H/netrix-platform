# 交付流程与质量标准

## 1. 交付原则

NeTrix 第一轮开发采用路径驱动，而不是页面驱动。每次迭代都应推进核心路径中的某个状态变化，例如完成档案引导、生成昵称建议、创建帖子、展示推荐、发送连接请求，或进入仅限已接受连接的消息线程。

团队不应长期停留在脚手架、视觉打磨或孤立组件阶段。基础设施完成后，应尽快形成一条真实、可点击、可持久化、可验证的端到端路径。

## 2. 分支与 PR 规则

不要直接在 `main` 上开发功能。分支命名使用：

```text
<github-login>/<short-task>
```

PR 应保持小而完整。一个 PR 可以完成一个纵向切片或一个清晰职责层变更，但不应同时大规模修改数据结构、界面、AI 提示词、RLS 和视觉系统。涉及 `SPEC.md`、数据结构、RLS、AI 输出结构或页面接口契约的 PR，必须在描述中标明下游影响。

PR 描述至少包含：

```text
Scope:
Validation:
Data/schema impact:
AI/security impact:
Remaining risk:
```

若验证命令无法运行，不能省略说明。应写清原因，例如依赖尚未安装、Supabase 本地实例未配置、OpenAI key 不存在，或端到端测试脚本尚未创建。

## 3. 三人开发流程

第一轮采用职责层拆分：

前端与产品流程负责人负责页面、布局、核心组件、状态反馈、响应式表现和界面文案一致性。

后端、数据与认证负责人负责 Supabase Auth、Drizzle 数据结构、RLS、服务端业务函数、权限辅助函数、种子数据和核心写入操作。

AI、推荐与质量负责人负责 AI 封装层、提示词版本、输出结构校验、推荐评分、事件记录、测试和核心路径验收。

三人并行时，数据结构、DTO、页面接口契约和 AI 输出结构是共享边界。任何成员需要改变这些契约，应先在 PR 描述中明确，并让相关成员评审。

## 4. 开工标准

一个开发任务进入实现前，必须具备以下条件：

- 对应产品行为能映射到帖子、档案、推荐、连接中的至少一环。
- 数据读写对象和权限边界明确。
- 前端状态至少覆盖相关的加载、空状态、错误或权限不足情况。
- 若涉及 AI，提示词版本、输入来源、输出结构和失败回退行为明确。
- 验收方式明确，可以通过测试、手动核心路径或演示种子数据验证。

不满足这些条件的任务应先补开发文档或拆小，而不是直接写代码。

## 5. 完成标准

一个功能完成必须同时满足产品、工程、质量和安全四个标准。

产品上，它必须推动当前闭环，而不是孤立增加功能。工程上，它必须包含前端状态、后端逻辑、数据持久化和错误处理。质量上，它必须通过相关验证命令，或明确记录无法验证的原因。安全上，它不得引入 API key 泄露、越权读取、真实个人数据暴露或不可审计的 AI 调用。

以下情况不能算完成：只有静态界面；后端写入成功但前端没有失败状态；AI 输出没有结构校验；消息权限只靠隐藏按钮；推荐无法解释生成依据；演示需要开发者手动修改数据库才能跑通。

## 6. 验证命令

网页脚手架建立后，包脚本应支持：

```bash
corepack pnpm lint
corepack pnpm typecheck
corepack pnpm test
corepack pnpm build
```

加入数据库和端到端测试后，应逐步支持：

```bash
corepack pnpm db:migrate
corepack pnpm db:seed
corepack pnpm test:e2e
```

每个 PR 只需运行与变更相关的最小验证，但合并到演示基线前必须跑完整验证家族。仅文档变更可以使用 `git diff --check` 和人工一致性审查作为主要验证。

## 7. 测试策略

单元测试优先覆盖会破坏主路径的纯逻辑：推荐评分、结构校验、权限辅助函数和 AI 输出解析。集成测试覆盖 server action 或 route handler 与数据库之间的状态转换，尤其是连接请求和消息权限。端到端测试至少覆盖一条核心路径：

```text
认证 -> 引导 -> 发帖 -> 推荐 -> 请求连接 -> 接受连接 -> 消息
```

第一版不追求漂亮的覆盖率数字，但不能缺少对核心权限和状态转换的验证。尤其是接受连接后才可消息，以及推荐记录这两类逻辑，不应只靠手动点击确认。

## 8. 第一轮迭代顺序

第一轮迭代建议按以下顺序推进。当前仓库已经完成第 1 步脚手架基线，后续应从第 2 步开始：

1. 已完成：建立 `apps/web` 脚手架，加入包脚本、Tailwind、基础布局和环境变量示例。
2. 后端负责人建立 Supabase/Drizzle 数据结构基线、认证辅助函数、档案和帖子表、RLS 草案。
3. 前端负责人建立路由骨架、信息流布局、引导流程、发帖组件和档案侧栏骨架。
4. AI 与质量负责人建立 AI 封装层骨架、提示词版本目录、推荐评分模块和校验测试。
5. 三人合并一条最小路径：演示用户、档案、帖子、推荐卡片、连接请求按钮。
6. 补充接受或拒绝连接、消息线程、事件记录、种子数据和核心路径验收。

这个顺序的核心原因是：越早跑通真实闭环，越早暴露产品和技术假设中的问题。NeTrix 当前最需要的是一个足够真实、稳定、可观察的 MVP，而不是抽象架构完整但无法使用的工程壳。

## 9. 演示门槛

进入正式演示前，团队必须能使用 3-5 个演示用户重复完成：校园邮箱用户注册或登录；档案引导；AI 昵称或档案建议；创建或浏览三类帖子；生成推荐档案；发送连接请求；对方接受；双方进入消息线程。

任何依赖隐藏脚本、手动数据库修改、跳过权限检查或真实学生个人数据的演示，都不能作为 MVP 演示。
