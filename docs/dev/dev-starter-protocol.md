# 开发启动协议

## 1. 目的

本文件定义 NeTrix 第一轮开发如何由三位开发者立即启动。它不是产品讨论文件，而是迭代操作协议。所有任务都应围绕当前 MVP 闭环推进：

```text
学术帖子 -> 学术档案 -> AI 辅助推荐 -> 学术连接
```

第一轮开发的目标是产出一个可运行的网页 MVP 脚手架，并尽快形成可演示的核心路径。团队不再重新选择技术栈，不再重开产品方向会议，也不再引入与当前闭环无关的大功能。

## 2. 角色分配

三人分工采用职责层拆分。每个成员都有清晰责任区，但必须共同维护共享契约。

| 角色 | 主要责任 | 必须协同的对象 |
| --- | --- | --- |
| 前端与产品流程负责人 | 路由、布局、组件、前端状态、响应式表现和界面文案。 | 与后端负责人确认 DTO 和操作接口；与 AI 负责人确认推荐与 AI 状态。 |
| 后端、数据与认证负责人 | Supabase Auth、Drizzle 数据结构、RLS、服务端业务函数、权限辅助函数和种子数据。 | 与前端负责人确认页面数据需求；与 AI 负责人确认推荐信号和事件记录。 |
| AI、推荐与质量负责人 | AI 封装层、提示词版本、输出结构、评分逻辑、测试和核心路径验收。 | 与后端负责人确认数据访问；与前端负责人确认界面状态和推荐解释展示。 |

如果团队实际能力分布不完全对应，可以调整人选，但不要改变这三个责任层。第一轮最重要的是减少架构争论和跨文件冲突。

## 3. 第 0 天准备

正式写代码前，三位成员应完成以下动作：

1. 阅读 `SPEC.md`、`docs/strategy/direction-memo.md`、`docs/dev/README.md`。
2. 各自阅读与自己角色相关的开发文件。
3. 确认本地工作区状态，避免覆盖其他人未提交变更。
4. 从 `main` 建立个人分支，命名为 `<github-login>/<short-task>`。
5. 将第一轮迭代任务拆成小 PR，不把脚手架、数据结构、界面、AI 和测试全部堆进一个 PR。

第 0 天不需要重新辩论产品定位。若发现文档与项目负责人最新方向冲突，先更新 `SPEC.md`，再继续实现。

## 4. 第一轮迭代产出

第一轮迭代的最低可交付成果如下。

前端与产品流程负责人应交付：

- `apps/web` 路由骨架和基础布局。
- `/auth`、`/onboarding`、`/feed`、`/posts/new`、`/posts/[id]`、`/me`、`/recommendations`、`/connections`、`/messages/[threadId]` 的页面骨架。
- `PostCard`、`PostComposer`、`AcademicProfileForm`、`RecommendationCard`、`ConnectionRequestButton`、`MessageThread` 的第一版组件。
- 加载、空状态、错误、权限不足、AI 生成中、请求待处理和已连接等基础状态。

后端、数据与认证负责人应交付：

- Supabase Auth 接入方案和校园邮箱验证保护。
- Drizzle 数据结构基线，覆盖用户、学术档案、帖子、推荐、连接请求、连接、消息、AI 任务和事件记录。
- RLS 策略草案，至少覆盖消息、连接请求、推荐和 AI 任务。
- 服务端认证辅助函数、权限辅助函数、档案、帖子、连接和消息服务。
- 覆盖 Math、CS、EEE 和三类帖子的演示种子数据。

AI、推荐与质量负责人应交付：

- `src/server/ai` 封装层骨架、提示词目录和输出结构。
- 昵称建议、学术画像、推荐解释的第一版提示词。
- 推荐资格过滤、候选生成和评分模块。
- AI 输出解析和失败回退行为。
- 事件记录辅助函数、核心单元测试和核心路径验收清单。

第一轮结束时，三人应合并出一条最小可运行路径，而不是三个彼此孤立的实现片段。

## 5. 共享契约评审

以下内容属于共享契约，任何变更都需要至少一个相关成员评审：

- Drizzle 数据结构和枚举。
- DTO、校验结构、server action 输入与输出。
- 路由契约和页面数据需求。
- AI 提示词版本、输入摘要、输出结构。
- RLS 策略和权限辅助函数。
- 推荐评分信号。

共享契约变更应在 PR 描述中单独写出，避免其他成员在不知情的情况下继续基于旧契约开发。

## 6. 日常协作节奏

第一轮开发建议采用轻量异步节奏：

```text
每日开始：说明当天目标和依赖。
开发过程中：契约变化立即写入 PR 描述。
每日结束：记录已合并内容、阻塞项和待评审内容。
```

不需要长会。若出现阻塞，应优先判断它属于产品决策、契约不一致、本地环境问题，还是实现错误。产品决策交给项目负责人；契约不一致通过小 PR 修正；本地环境问题写入开发文档；实现错误在当前分支解决。

## 7. PR 大小与合并顺序

推荐合并顺序：

1. 脚手架基线：包脚本、目录结构、Tailwind、环境变量示例。
2. 数据与认证基线：Drizzle 数据结构、Supabase client、认证辅助函数。
3. 路由与组件基线：路由骨架、布局、核心界面组件。
4. 档案与帖子切片：引导流程、档案持久化、发帖和信息流。
5. AI 与推荐切片：评分、AI 解释、推荐卡片。
6. 连接与消息切片：请求、接受、拒绝、仅限已接受连接的消息。
7. 加固：测试、种子数据、事件记录、移动端布局、演示清理。

PR 越早合并越好，但不能牺牲契约清晰度。一个 PR 如果让其他成员无法判断自己该如何继续，应拆小或补充文档。

## 8. 阻塞处理

阻塞处理原则如下：

- 技术栈不明确：以 `SPEC.md` 和 `docs/dev` 为准，不重新选择。
- 产品不明确：由项目负责人决定；决定后更新 `SPEC.md`。
- 数据结构不明确：后端负责人提出方案，AI 与前端负责人评审数据需求。
- 界面不明确：前端负责人提出方案，确保不破坏产品流程和权限语义。
- AI 不明确：AI 负责人提出方案，确保不越过 AI 边界和隐私规则。

不要用临时手段绕过权限、AI 校验或数据模型。第一版可以小，但不能留下会破坏信任的基础问题。
